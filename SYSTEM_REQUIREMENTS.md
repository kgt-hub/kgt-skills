## kdb 要件

### 目的
過去〜現在のメールを資産として活用し、業務の自動化・意思決定支援を行う。

### メール取込み方針
- **ソース**: Gmail Business Starter（共有受信箱）
- **取込み方法**: Gmail API（本文はサーバー内で処理・外部送信なし）
- **対象**: 全受信メール（常時取込み）
- **言語**: タイ語・英語・日本語 混在

### 分類・処理フロー
```
Gmail 共有受信箱
    │ Gmail API（取得のみ）
    ↓
kdb（192.168.1.101 内で完結）
    │
    ├─ ローカル LM で分類
    │   引き合い / PO / クレーム / 問い合わせ / その他
    │
    ├─ 担当者自動割り当て
    │   ルール: 得意先固定担当が優先 → 未登録は種別で振る
    │
    ├─→ titan: 引き合い候補を「確認待ち」で仮登録
    │          担当者が「登録 / 却下」を1クリックで判断
    │
    ├─→ order-control: PO 添付メールを受注候補として仮登録
    │
    └─ 通知: システム内ダッシュボードに「未処理 X件」表示
             ※ システムからのメール自動送信は禁止
             ※ 担当者が確認後に自分で Gmail から返信
```

### 誤分類への対応方針
- 「緩めに取り込んで、引き合いでなければ却下」
- 仮登録 → 人間が1クリックで承認/却下
- 全自動ではなく「確認付き自動」

### データ層の設計
```
【生データ層】完全機密 - kdb のみ・MD 専用
  メール本文・得意先名・価格・引き合い詳細

【集計・分析層】社内公開
  得意先別受注率・外注先納期遵守率・価格推移
  営業 KPI・購買 KPI

【知見層】社内公開（MD 承認後）
  業種別引き合い傾向・アピール商材・営業ヒント
  ※ 得意先名・価格は含めない・業種×商材の粒度
```

### 戦略データ（営業・購買）
**営業面:**
- 得意先別の引き合い頻度・受注率
- 失注理由の傾向（価格/納期/品質）
- 季節性・繰り返し発注パターン
- 得意先との最終接触日（フォローアップ漏れ検知）
- 顧客業種×引き合い商材の傾向 → 新規営業の共通知

**購買面:**
- 外注先・材料の価格推移
- 仕入先ごとの納期遵守率・品質クレーム頻度
- 相見積の勝敗パターン

### 機密情報の扱い
- 社内公開: OK（集計・知見層）
- 社外 AI 送信: 禁止（ローカル LM で処理）
- 知見層の公開: MD 承認後のみ

## システム連携方針

### titan ↔ order-control 連携（最優先設計テーマ）
```
現状（問題）:
  titan で受注確定 → 手動転記 → order-control に受注登録
  → 二重入力・転記ミスのリスク

理想:
  titan で「受注確定」→ order-control に自動で受注データ作成
```

### kdb → titan / order-control 連携
```
引き合いメール → kdb → titan に「確認待ち」仮登録
PO メール     → kdb → order-control に「確認待ち」仮登録
```
