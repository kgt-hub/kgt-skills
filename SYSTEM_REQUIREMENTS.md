- order-control: `http://192.168.1.101/order`（port 8501）

### AI 処理方針（重要）
- **社外 AI へのデータ送信禁止**（機密情報保護）
- **ローカル LM（LM Studio）を使用する**（titan で実績あり）
- Gmail API はメール取得のみ（本文は外部送信しない）
- システムからのメール自動送信禁止（担当者確認後に自分で送信）

- マシンスペックにより使用モデルは流動的・設計はモデル非依存にする

### スタッフ公開に向けた障壁
1. 要件をまさに具体化中（本文書がその作業）
2. スタッフの IT リテラシーが低い（操作ミス・入力ミス・迷子になるリスク）

**対応方針（IT リテラシー）:**
- 1画面にやることを1つに絞る（複数タブ・複数操作を同時に見せない）
- エラーメッセージをわかりやすいタイ語で表示
- 間違えても取り消せる設計（削除は Manager のみ・確認ダイアログ必須）
- 初期トレーニング用のシンプルなマニュアル（画像多め・文字少なめ）

### 業務フロー全体像（2026-04-10 確定）

kdb が業務の入口となり、全システムがつながる。

```
【上流】
Gmail 共有受信箱にメール着信
    ↓
kdb: 分類・担当自動割り当て
    ↓
titan: 引き合い仮登録（確認待ち）→ 担当者が承認
    ↓
titan: 見積作成・提出
    ↓
titan: 受注確定 → order-control に自動登録
    ↓
【下流】
order-control: 工程進捗（生産）
    ↓
order-control: 品質検査（品質担当）
    ↓
order-control: 出荷確認（物流）
    ↓
order-control: 売上・粗利確認（MD）
```

**kdb は「後から追加するシステム」ではなく全体の入口。**
kdb が整わないと全フロー検証ができない。

### 運用テストの原則（重要）
データは上流から下流まで業務フロー全体を通して流れる。
部分的な公開・テストでは前後工程のデータがなく、間違いに気づけない。
**運用テストは全担当ロールが揃った状態で全フローを通して実施する。**

- 一斉公開を前提とする（ロール別の段階公開はしない）
- 公開前に MD が全ロールを演じてフルフロー検証を完了させる
- トレーニングは全員同時・フロー順に実施する
- **kdb の完成が全体公開のボトルネック**

### メール取込み方針
- **ソース**: Gmail Business Starter（共有受信箱: m_iwasaki@thai-konan.com）
- **取込み方法**: Gmail API（本文はサーバー内で処理・外部送信なし・既読変更なし）
- **対象**: 全受信メール（常時取込み）
- **言語**: タイ語・英語・日本語 混在
- **頻度**: 15分ポーリング予定（**cron 設定は未実施・後で対応**）

### 実装済み（2026-04-10）
- `gmail_auth.py`: OAuth2 認証・token.json 管理
- `gmail_sync.py`: 増分取込み（`--days=N` で日数指定可、通常は前回以降を自動取得）
- `classify.py`: メール分類ルーチン v2（方向性判定 + LM 詳細分類）
- `emails_dedup` ビュー: mbox と gmail_api の重複除去（gmail_api 優先）

### DB 現状（2026-04-10）
| sync_source | 件数 |
|---|---|
| mbox（3年分） | 36,966件 |
| gmail_api（直近7日） | 424件 |
| emails_dedup（重複除去後） | 37,219件 |

### 分類・処理フロー
```
Gmail 共有受信箱
    │ Gmail API（取得のみ）
    ↓
kdb（192.168.1.101 内で完結）
    │
    ├─ ローカル LM で分類
    │   引き合い / PO / クレーム / 問い合わせ / その他
    │
    ├─ 担当者自動割り当て
    │   ルール: 得意先固定担当が優先 → 未登録は種別で振る
    │
    ├─→ titan: 引き合い候補を「確認待ち」で仮登録
    │          担当者が「登録 / 却下」を1クリックで判断
    │
    ├─→ order-control: PO 添付メールを受注候補として仮登録
    │
    └─ 通知: システム内ダッシュボードに「未処理 X件」表示
             ※ システムからのメール自動送信は禁止
             ※ 担当者が確認後に自分で Gmail から返信
```

### 誤分類への対応方針
- 「緩めに取り込んで、引き合いでなければ却下」
- 仮登録 → 人間が1クリックで承認/却下
- 全自動ではなく「確認付き自動」

### データ層の設計
```
【生データ層】メール本文・添付・価格・値引き詳細
【集計・分析層】価格推移・値引き傾向・受注率・KPI
【知見層】業種別傾向・アピール商材・過去仕様・図番検索
```

### アクセス権限方針
- **現状（PC操作担当 7名）**: 全員 Manager 権限で全層を開放
  → 価格・値引き・採算情報も含め現状を把握させ当事者意識を持たせる
- **将来（大人数化時）**: ロール分離できる設計にしておく
  - General: メール本文・添付・仕様・図面検索
  - Manager: 上記 ＋ 価格推移・値引き傾向・採算データ

### kdb ナレッジベース機能（スタッフ向け）
以下を横断検索できる UI を提供する：

**A: メール・添付の直接参照**
- 取引先・キーワード・期間・ファイル種別でフィルタ
- 意味検索（タイ語/英語/日本語 混在でも有効・埋め込みモデル活用）
- メール本文・添付ファイルを開ける

**C: 図番・仕様の検索**
- 図番・材料・寸法で過去案件を検索
- CAD ファイル（.sldprt/.step 計329件）の存在確認
- titan の知識DB と連携

**価格・値引き傾向（現状全員・将来 Manager 権限）**
- 得意先別の過去提示価格推移
- 値引き依頼の傾向・受諾率
- 採算ラインの感覚値を組織知として共有

### 戦略データ（営業・購買）
**営業面:**
- 得意先別の引き合い頻度・受注率
- 失注理由の傾向（価格/納期/品質）
- 季節性・繰り返し発注パターン
- 得意先との最終接触日（フォローアップ漏れ検知）
- 顧客業種×引き合い商材の傾向 → 新規営業の共通知

**購買面:**
- 外注先・材料の価格推移
- 仕入先ごとの納期遵守率・品質クレーム頻度
- 相見積の勝敗パターン

### 機密情報の扱い
- 社内公開: 全層 OK（現状は全員 Manager 権限）
- 社外 AI 送信: 禁止（ローカル LM で処理）
- 知見層の公開: MD 承認後のみ

## システム連携方針

### titan ↔ order-control 連携（最優先設計テーマ）
```
現状（問題）:
  titan で受注確定 → 手動転記 → order-control に受注登録
  → 二重入力・転記ミスのリスク

理想:
  titan で「受注確定」→ order-control に自動で受注データ作成
```

### kdb → titan / order-control 連携
```
引き合いメール → kdb → titan に「確認待ち」仮登録
PO メール     → kdb → order-control に「確認待ち」仮登録
```
